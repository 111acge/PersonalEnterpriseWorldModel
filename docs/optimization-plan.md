# 个人企业世界模型（PEWM）后续优化方案

## 一、当前 baseline 诊断

### 1.1 项目现状

PEWM 已完成功能：

| 模块 | 状态 |
|------|------|
| Flask + pywebview 桌面 UI | 已上线，含启动画面 |
| 本地 SQLite 数据层 | 已迁移，支持文档/对话历史 |
| 向量检索（TF-IDF / bge） | 已支持，SQLite 持久化 |
| AI 管线（Inbox → 实体提取 → 索引） | 已支持，含 watchdog 自动触发 |
| RAG 对话 | 已支持，含多轮历史 |
| 配置管理（LLM/OCR/用户/提示词） | 已支持 |
| 基础测试 | 14 个用例通过 |

### 1.2 核心指标与瓶颈

| 维度 | 当前 baseline | 核心痛点 |
|------|--------------|---------|
| 包体积 | ~288 MB | torch/transformers/scikit 占大头，分发困难 |
| 启动时间 | 5–15 秒（本地模型已存在时 3–5 秒） | 模型加载、Flask/向量库初始化阻塞启动 |
| 运行时内存 | ~1.5–3 GB | torch + transformers 常驻内存高 |
| 检索延迟 | <100 ms（TF-IDF）；200–500 ms（bge 向量） | 大文档多后检索变慢 |
| AI 管线成本 | 每文件一次 LLM 调用 | 批量提取仍有优化空间 |
| 测试覆盖 | 14 个用例 | 未覆盖 pipeline、RAG、前端 |
| 用户体验 | 窗口无边框，无原生拖动/最小化 | 操作体验不够原生 |

### 1.3 关键瓶颈定位

1. **embedding 模型过重**：`bge-small-zh` 约 100MB，torch 运行时占 200MB+，导致包体积和内存双高。
2. **启动链路过长**：配置 → 数据库 → 向量库 → 模型 → Flask 串行加载，任一环节慢都影响启动。
3. **前端仍是 SPA**：pywebview 单窗口，缺少原生窗口操作（最小化、最大化、拖拽标题栏）。
4. **缺少指标监控**：无日志、无性能埋点、无崩溃上报。
5. **LLM 调用未做流式/异步**：RAG 回答等待时间长，UI 无打字机效果。
6. **向量索引无增量策略**：每次重建向量索引成本高。

## 三、具体落地步骤

### 3.1 启动与体积瘦身

1. **模型可选化**
   - 将 `sentence-transformers` + `torch` 改为可选依赖。
   - 默认使用纯 Python/NumPy 的轻量 embedding（如 `text2vec` 或自研 tiny embedding）。
   - 提供「完整版」和「轻量版」两个 build target。

2. **启动并行化**
   - 在 `SplashController` 中用线程池并行执行：读配置、初始化 DB、初始化向量库。
   - 模型加载改为后台预加载，首次检索时才真正初始化。

3. **按需导入**
   - 在 `rag.py`、`retrieval.py` 中延迟导入 transformers/torch，避免启动时加载。

4. **PyInstaller 精简**
   - 用 `--exclude-module torch` 构建轻量版。
   - 移除未使用的 sklearn/scipy 子模块。

### 3.2 交互体验升级

1. **流式输出**
   - 后端：`/api/chat` 改为 SSE（Server-Sent Events）流式返回。
   - 前端：监听 SSE，逐段渲染回答。

2. **原生窗口壳**
   - 给主窗口加自定义标题栏（HTML/CSS 实现），支持：
     - 鼠标按住标题栏拖动窗口
     - 最小化、最大化/还原、关闭按钮
   - 调用 `pywebview` 的 `window.move()` / `window.resize()` API。

3. **主题与动画**
   - 全局 CSS 变量统一，移除硬编码颜色。
   - 页面切换、弹窗、Toast 增加过渡动画。

### 3.3 检索与管线效率

1. **增量向量更新**
   - 记录每篇文档的 `content_hash` 和 `indexed_at`。
   - 管线运行时只计算新增/变更文档的向量，删除文档的向量软删除。

2. **检索缓存**
   - 对最近 100 条查询做 LRU 缓存，缓存键为 `query+entity_type`。

3. **LLM 批量提取**
   - 已初步实现 `_llm_extract_batch`，后续：
     - 控制单批次 token 数（< 8K）。
     - 对批量结果做后校验，缺失字段自动单文件兜底。

### 3.4 稳定性与可观测性

1. **日志体系**
   - 引入 `loguru` 或标准 logging，分级别写入 `logs/app.log`。
   - 启动、检索、LLM 调用、异常均记录。

2. **性能埋点**
   - 在关键函数加装饰器：
     - `start_desktop_app` 总启动时间
     - `VectorDB.search` 检索耗时
     - `rag_answer` 端到端耗时
   - 提供 `/api/metrics` 接口供调试。

3. **测试扩展**
   - 新增 `tests/test_pipeline.py`：模拟 Inbox 文件处理全流程。
   - 新增 `tests/test_rag.py`：mock LLM 测试 RAG 组装逻辑。
   - 前端：引入 Vitest + jsdom 测试 app.js 核心逻辑。

## 四、验证机制

### 4.1 A/B / 对照测试

| 优化项 | 对照组 | 实验组 | 指标 |
|--------|--------|--------|------|
| 轻量版 exe | 完整版 288MB | 轻量版 <100MB | 启动时间、包体积、功能可用率 |
| 启动并行化 | 串行启动 | 并行启动 | 启动总耗时 |
| 流式输出 | 一次性返回 | SSE 流式 | 首 token 时间、用户主观满意度 |
| 增量向量 | 全量重建 | 增量更新 | 管线耗时、CPU 占用 |

### 4.2 灰度发布

- 内部先跑 1–2 周，收集日志。
- 发布到 GitHub Release，标记 `beta`/`stable` 两个 channel。
- 用户可回退到上一稳定版。

### 4.3 指标监控

- 本地 SQLite 表 `metrics` 记录：
  - `event`（启动/检索/对话/管线）
  - `duration_ms`
  - `success`
  - `error_msg`
- 设置页增加「诊断信息」面板，展示最近 100 条指标。

## 五、风险与应急预案

| 风险 | 影响 | 规避/应急 |
|------|------|----------|
| 剥离 torch 后语义检索质量下降 | 检索精度降低 | 保留完整版下载入口；轻量版用 TF-IDF + 小模型 |
| SSE 流式改造引入连接不稳定 | 对话中断 | 保留非流式 fallback 接口 |
| 增量向量逻辑 bug 导致索引不一致 | 检索不到新内容 | 提供「重建向量索引」按钮；定期全量校验 |
| 新窗口壳导致拖动/关闭失效 | 用户体验倒退 | 保留旧版 UI 配置项；快速回滚 |
| 日志文件无限增长 | 磁盘占满 | 日志按天轮转，保留 7 天 |
| 测试不足引发回归 | 核心功能损坏 | 每次提交跑 pytest；PR 必须全绿 |
